# Процесс Разработки Проекта "niNE"

Этот документ описывает текущее состояние проекта, планы на будущее и правила внесения изменений в кодовую базу.

## Текущий Статус Проекта

### Реализовано

На данный момент в проекте реализован следующий основной функционал:

-   **Клиент-серверная архитектура**: Основа проекта, использующая `asyncio` для асинхронной работы и SSL для защищенного соединения.
-   **База данных**: Сохранение данных игроков (позиция, имя) в базе данных SQLite.
-   **Система аккаунтов**: Базовая регистрация и аутентификация игроков.
-   **Игровой мир**:
    -   Основной мир с плоскостью земли.
    -   Синхронизация передвижения, вращения и анимаций персонажей между клиентами в реальном времени.
-   **Персонаж и Управление**:
    -   Модель игрока с анимациями ходьбы и простоя.
    -   Управление персонажем с клавиатуры.
    -   Улучшенная камера от третьего лица (mouselook) с возможностью вращения.
-   **Пользовательский интерфейс (UI)**:
    -   Главное меню, меню входа, игровое меню.
    -   Меню настроек с возможностью изменять никнейм, разрешение экрана и чувствительность камеры.
    -   Игровой чат с отправкой и получением сообщений.
-   **Система плагинов**: Архитектура проекта поддерживает модульные плагины для расширения функционала.

### В Разработке

Следующие большие задачи находятся в планах или на ранней стадии разработки:

-   Боевая система (умения, урон, здоровье).
-   Система инвентаря и предметов.
-   Добавление NPC и квестов.
-   Расширение и усложнение игрового мира.
-   Улучшение и доработка пользовательского интерфейса.

## Процесс Внесения Изменений (Contribution Workflow)

Для поддержания порядка в репозитории и упрощения процесса разработки все изменения должны вноситься согласно следующим правилам:

1.  **Основная ветка для разработки - `dev`**. Ветка `main` содержит только стабильные, протестированные версии. Весь новый функционал и исправления добавляются сначала в `dev`.

2.  **Клонирование ветки**: Перед началом работы над новой задачей необходимо склонировать актуальную версию ветки `dev`.

3.  **Создание MD-файла с описанием**: Для каждого значимого обновления (новая функция, крупное исправление) необходимо создать небольшой `.md` (Markdown) файл, в котором кратко описывается суть изменений. Это помогает другим разработчикам быстрее понимать, что было сделано.

4.  **Слияние (Merge) в `dev`**: После завершения работы и написания `.md` файла, ваша ветка с изменениями должна быть слита (merged) в ветку `dev`.

5.  **Слияние в `main`**: Слияние из ветки `dev` в `main` выполняется **только** главным разработчиком (мной) после проверки и тестирования. **Никогда не делайте merge в `main` самостоятельно.**
